# Runbook — "A new game has been chosen for a meeting"

Follow this checklist whenever a new game has been picked for meeting **#N** (usually with a Discord
announcement: game title, date, time, venue, banner). The homepage upcoming-events and history sections
are **database-backed**, so this is now **admin data entry**, not HTML editing. You enter the meeting and
its selected game through `vote-admin.html`; the shared renderer (`js/meetings.js`) builds the event card,
history card, calendar links, platform icons, and JSON-LD from D1 for both languages.

The per-field details live in [`docs/content-guide.md`](docs/content-guide.md); the rounds/meetings data
model lives in [`docs/voting-system.md`](docs/voting-system.md).

---

## Prerequisite — production D1 must hold the meeting backfill (one time)

The live homepage only renders from D1 once the production database has meeting rows. Until then the static
fallback cards in `index.html` / `en/index.html` are what visitors see. Backfill production **once** before
relying on the database path (see [`docs/deployment-guide.md`](docs/deployment-guide.md) "D1 Migration"):

```powershell
wrangler d1 execute gamestormers --remote --file=./backfill-meetings.sql
```

`backfill-meetings.sql` is generated by `scripts/build-backfill-sql.mjs` from the historical static cards.
Past meeting dates in it are approximate (first Monday of the month) and can be corrected later via the admin.

## 0. Inputs to collect from the announcement

| Field | Default if not stated |
|---|---|
| Meeting number **N** | — |
| Game title | — |
| Date | "Typically the first Monday of the month" — confirm the weekday |
| Time | **18:30–~21:00** (start 18:30, ~21:00 estimated end) |
| Venue | **Folkehuset Møllestien, Grønnegade 10, 8000 Aarhus C** — only change if the announcement differs |
| Discord | General invite `discord.gg/N2h6DJxVDF` (used in calendar details) |

## 1. Verify the game on Steam (don't trust the title alone)

- Find the **Steam App ID** — the ID in the store URL is authoritative. Wrong/duplicate IDs are common.
- **Sanity-check it's the right game**: titles collide. Compare the announcement's banner art and premise
  against the Steam page before committing to an ID.
- Note the **platforms** Steam reports (Windows / macOS / Linux). Include only the ones marked true.
- Check **GOG**: include a GOG link only if a GOG page actually exists. For sale-badge matching you also need
  the numeric **GOG product id**.
- Read the Steam premise/genres to write **accurate** copy. If the announcement conflicts with the real game,
  lean on Steam and flag the mismatch to the maintainer.

## 2. ⚠️ Ask the maintainer for the HowLongToBeat link

HowLongToBeat blocks automated fetching, so the canonical `howlongtobeat.com/game/{id}` link and the playtime
**cannot be looked up automatically — always ask the maintainer for the specific link and the hours.**
Round to whole hours to match house style (e.g. 9.5 h → `~10 t.` / `~10 hrs.`).

## 3. Make sure the round and its meeting record exist

Open `vote-admin.html` (local: use the `ADMIN_TOKEN` from `.dev.vars`). A voting round and a public meeting
record share the same id = meeting number.

- If the round was created through the normal voting cycle, opening it already created the matching `meetings`
  row with the meeting date, Copenhagen-local start/end times, venue, and Discord invite.
- If you are entering a meeting that never went through voting, create/repair the round's event basics in the
  round editor (meeting date, start/end time, venue, Discord). The admin shows whether a public meeting record
  exists for the round.

## 4. Promote the chosen game into the meeting

In the round's **Selected game** section of `vote-admin.html`:

- Pick the winning suggestion (most-voted first) and **promote** it. This copies the suggestion into a `games`
  row, attaches it to the meeting (`selected_game_id` / `selected_suggestion_id`), confirms
  `rounds.winner_suggestion_id`, and reveals the round when appropriate.
- Non-Steam suggestion approval stays manual; promotion does not change a suggestion's approval status.

## 5. Fill in the maintainer-only fields

Still in the Selected game editor, complete the fields the homepage card needs. The **publish-readiness**
banner tells you what is still missing; a card is not publish-ready until the game has a title, cover image,
store link, genres, platforms, playtime hours, HowLongToBeat URL, and **Danish + English event descriptions**.

- GOG URL and numeric GOG product id (for sale-badge matching).
- HowLongToBeat URL and whole-hour playtime.
- Genres and platforms (correct Steam import if needed).
- Danish and English **event** descriptions (paragraphs separated by a blank line).
- Optional Danish and English **history** descriptions (short blurbs). If omitted, the history card falls back
  to the event description.

## 6. Verify in a real browser

Run the Pages dev server and check the rendered output (never just raw files):

```powershell
npm run dev
```

- `http://127.0.0.1:8788/` and `/en/`: the new meeting shows as an upcoming event card, with correct meeting
  number, banner, platform icons, genres, playtime link, DST-correct time range, calendar dropdown, and
  localized description. The history count is right. JSON-LD has the regenerated `Event` for the meeting.
- `http://127.0.0.1:8788/vote.html` and `/vote-admin.html`: no console errors.
- Confirm the previous meeting moves into history on its own after its end time (no manual reveal step).

## 7. Update `sitemap.xml`

Set `lastmod` to today's date on **both** URL entries for meaningful content changes.

## 8. Do **not** commit or push unless explicitly asked

Pushing to `main` **auto-deploys to the live site**. Leave commit/push to the maintainer's go-ahead. Note that
meeting content lives in D1, not in the repo, so publishing a meeting to the live site means writing to the
**production** database (through the deployed `vote-admin.html` or a `--remote` D1 command), not committing HTML.

## Legacy static-HTML fallback (transition only)

The static event/history cards and their JSON-LD remain in `index.html` / `en/index.html` as a no-JS and
empty-database fallback. They are not the live content path once production D1 is backfilled. If you ever need
to edit them during the transition, keep Danish and English in sync and mirror the database fields; the
component markup is documented in [`docs/content-guide.md`](docs/content-guide.md). Removing these duplicated
static cards is deferred until the production backfill is confirmed, so the live site can never render empty.
